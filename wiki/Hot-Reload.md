# Hot Reload

Meta-C features **native hot reload** — you can modify source code, recompile a `.so`, and swap it into a running program without restarting.

---

## How It Works

Hot reload is built on three Linux primitives:

| Primitive | Role |
|-----------|------|
| **dlopen** | Dynamically load `.so` shared libraries at runtime |
| **dlsym** | Look up function symbols in the loaded library |
| **inotify** | Monitor filesystem for changes to `.so` files |

The architecture is straightforward:

```
1. Program starts with hot reload engine
2. Engine loads initial .so (dlopen)
3. You edit code, recompile → new .so appears
4. inotify detects the file change
5. Engine copies the .so (to bypass dlopen cache)
6. dlopen loads the copy
7. Function pointers are atomically swapped (__atomic_store)
8. Old .so is closed (dlclose) — but only after swap is complete
```

During the swap, all block allocations are **frozen** (spin-wait) to prevent corruption. The freeze lasts only for the duration of `dlopen`, typically <1ms.

## Setup

### Project Structure

Each package that needs hot reload becomes its own `.so`:

```
game/
├── src/           → your .mc files
├── build/
│   ├── libgame.so     ← the hot-reloadable library
│   └── game.c         ← generated C code
```

### Compilation Steps

```bash
# 1. Compile .mc → .c
meta-c game.mc -o build/game.c

# 2. Compile C → .so (position-independent code)
gcc -O3 -shared -fPIC -o build/libgame.so build/game.c runtime/block_memory.c runtime/io.c -ldl

# 3. Link main program (with hot reload support)
gcc -O3 main.c runtime/hot_reload.c runtime/block_memory.c runtime/io.c -o game -ldl -rdynamic
```

### Runtime Setup

```c
#include "hot_reload.h"

int main() {
    // 1. Create the hot reload engine
    HotReloadEngine* hr = hr_create("build/libgame.so");

    // 2. Register function pointers to be hot-swapped
    //    (generated by meta-c compiler)
    hr_register_func(hr, "game_update", (void**)&game_update);
    hr_register_func(hr, "game_render", (void**)&game_render);

    // 3. Load initial symbols
    hr_load_initial(hr);

    // 4. Start file monitoring (creates a background thread)
    hr_start_watching(hr);

    // 5. Game loop — uses function pointers that get swapped
    while (running) {
        game_update(dt);
        game_render();
    }

    // 6. Cleanup
    hr_destroy(hr);
    return 0;
}
```

## Using Hot Reload

### Typical Workflow

```bash
# Terminal 1: Run the program
./game

# Terminal 2: Edit code, recompile
vim game.mc                         # edit
meta-c game.mc -o build/game.c      # compile .mc → .c
gcc -O3 -shared -fPIC -o build/libgame.so build/game.c runtime/*.c -ldl

# ← Program detects change automatically, swaps code
# ← You see the new behavior immediately!
```

### What Gets Swapped

- **Function implementations** — new code runs on next call
- **Data** — existing data in blocks is preserved
- **Struct layouts** — must not change (ABI stability)

### What Does NOT Get Swapped

- **Global variables** — these stay with the main program
- **Block declarations** — blocks are allocated by the main program
- **Struct/class definitions** — memory layout must remain compatible

## Engine API Reference

`runtime/hot_reload.h`:

```c
// Create a hot reload engine for a .so library
HotReloadEngine* hr_create(const char* so_path);

// Load initial symbols from the .so
int hr_load_initial(HotReloadEngine* hr);

// Register a function pointer for hot swapping
// name: symbol name in the .so
// func_ptr: address of a void* that will be updated on reload
int hr_register_func(HotReloadEngine* hr, const char* name, void** func_ptr);

// Start watching the .so file for modifications (inotify)
// Creates a separate monitoring thread
int hr_start_watching(HotReloadEngine* hr);

// Force a reload immediately
int hr_reload(HotReloadEngine* hr);

// Get current state
HotReloadState hr_state(HotReloadEngine* hr);

// Set a callback that fires after every reload attempt
void hr_set_callback(HotReloadEngine* hr, hr_callback_t cb);

// Stop monitoring and cleanup
void hr_destroy(HotReloadEngine* hr);
```

### States

```c
typedef enum {
    HR_WAITING,   // Not yet loaded
    HR_LOADING,   // Reload in progress
    HR_OK,        // Loaded successfully
    HR_ERROR,     // Failed to load
} HotReloadState;
```

## Internal Details

### Atomic Function Pointer Swap

When a reload happens, function pointers are swapped using atomic operations:

```c
// From hot_reload.c — atomic store of new function pointer
__atomic_store(hr->symbols[i].func_ptr, &sym, __ATOMIC_SEQ_CST);
```

This ensures that any thread calling the function sees either the old version or the new version — never a torn pointer.

### Block Freeze During Reload

During the swap, `block_freeze()` causes `block_alloc()` to spin-wait:

```c
// From block_memory.c — spin loop during freeze
while (atomic_load_explicit(&block_frozen_flag, memory_order_acquire)) {
    __asm__ volatile("pause");  // CPU hint — yields to hyperthread
}
```

The freeze duration is typically microseconds (just the time to call `dlopen` and swap pointers).

### Cache-Busting with Temp Files

`dlopen` caches loaded libraries by path. If you write a new `.so` to the same path, `dlopen` returns the old cached handle. Meta-C's workaround:

```c
// Copy .so to a temp file, dlopen the copy, delete it
// Copying also ensures the file is fully written
snprintf(tmp_path, sizeof(tmp_path), "%s.%d",
         hr->so_path, (int)time(NULL));
// ... copy file ...
new_handle = dlopen(tmp_path, RTLD_NOW | RTLD_LOCAL);
unlink(tmp_path);
```

### Rollback on Failure

If a reload fails (e.g., compilation error in the new `.so`), the old code remains active:

```c
if (!new_handle) {
    // Keep old handle — function pointers untouched
    hr->state = HR_OK;  // or HR_ERROR if no handle exists
    block_thaw();
    return -1;
}
```

## Best Practices

### 1. Keep Data Layout Stable

Hot reload only swaps **code**, not data. If you change a struct's field layout, the old data becomes incompatible. Guidelines:

- Add new fields **at the end** of structs
- Never remove fields
- Never reorder fields
- Never change field types

### 2. Minimize Global State

Function-level hot reload works best when functions are stateless or use block-allocated data:

```meta-c
// Good: function reads from parameters, no global state
fn update_player(Player p, float dt) {
    p.hp -= dt * p.damage_rate
}

// Bad: function relies on global that can't be hot-swapped
fn global_update() {
    game_state.tick += 1  // game_state is in main's block
}
```

### 3. Use a Game Loop Pattern

```meta-c
fn main() {
    // Blocks are declared here, persist across reloads
    block global = 256MB
    block temp = 8MB

    // Initialization
    Player p = Player(100, "Felipe", 30) @global

    while running {
        temp.reset()                 // clear per-frame data

        // Hot-reloadable update
        p.update(delta_time) @temp   // uses block scope

        // Render
        render(p)
    }
}
```

### 4. Handle Errors Gracefully

Set a callback to be notified of reload attempts:

```c
void on_reload(const char* so_path) {
    HotReloadState state = hr_state(hr);
    if (state == HR_OK) {
        printf("✓ Hot reload of %s successful\n", so_path);
    } else {
        printf("✗ Hot reload failed — using previous code\n", so_path);
    }
}

hr_set_callback(hr, on_reload);
```

## Limitations

| Limitation | Reason | Workaround |
|-----------|--------|------------|
| Linux only | Uses inotify, dlopen | — |
| .so per package | Each package compiles independently | Structure your packages well |
| No data hot reload | Only code symbols are swapped | Design with this in mind |
| ABI stability | Struct layouts must not change | Add fields at end only |
| Thread safety | Function pointers are atomic, but data is not | Use your own synchronization |
| No Windows support | dlopen + inotify are POSIX | Limited to Linux (for now) |
